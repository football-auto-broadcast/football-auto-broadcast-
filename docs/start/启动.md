# football-auto-broadcast 系统启动说明

**最后更新**: 2026-06-27（实机联调修正版）
**适用版本**: develop 分支 (C 模块 FFmpeg 子进程方案)

---

## 1. 系统架构概览

```
GigE相机 → A(采集推流) → MediaMTX → RTSP ─┬─► B(录像) → raw/cut/program MP4
              8554/main, 8555/aux           ├─► B(preview) → 8560/program
                                            ├─► C(视觉分析) → focus_region / event / decision
                                            ├─► D(集锦生成) → full_highlight.mp4
                                            └─► E(平台调度) → 全流程控制
```

### 模块端口速查

| 模块 | 可执行文件 | HTTP | RTSP/用途 |
|------|-----------|------|-----------|
| **A** 采集推流 | `services/ingest-streaming-module/bin/ingest-streaming-module.exe` | `8081` | GigE 相机 → RTSP 8554/main, 8555/aux |
| **MediaMTX ×3** | `services/ingest-streaming-module/bin/mediamtx.exe` | — | 8554, 8555, **8560** |
| **B** 录像 | `services/record-program-module/build/record_program_service.exe` | `8082` | RTSP → raw/cut/program MP4; preview → 8560/program |
| **C** 视觉分析 | `services/vision-event-module/build-ninja-msvc/vision_event_service.exe` | `8083` | 双路帧 → focus_region / event / decision |
| **D** 集锦 | `services/highlight-generation-module/build/highlight_generation_service.exe` | `8084` | 事件 + 录像 → 集锦 MP4 |
| **E** 平台调度 | `services/platform-orchestration-module/build/bin/platform_orchestration_service.exe` | `8080` | 全流程编排 + Web UI：<http://127.0.0.1:8080/> |

### 平台前端网址

E 模块启动成功后，在本机浏览器打开：

**<http://127.0.0.1:8080/>**

该页面是平台操作前端，可用于创建、启动、停止比赛，查看各模块状态并下载集锦。前端由 `platform_orchestration_service.exe` 提供，因此 E 模块未启动时该网址无法访问。

## 3. 启动步骤

打开 **4 个 PowerShell 终端**，严格按照顺序启动。

### 终端 1 — A 模块（采集推流，必须最先启动）

```powershell
# 进入 A 模块 bin 目录
cd I:\GitHub\football-auto-broadcast-\services\ingest-streaming-module\bin

# 步骤 1: 启动 MediaMTX 三实例（8554/main、8555/aux、8560/program）
# ⚠️ 8560 必不可少！B 模块的 program preview 需要它，缺失会导致 B stop 挂死
# 注意：不能直接双击 mediamtx.exe 三次。每个实例必须指定不同的配置文件，
# 否则三个进程都会尝试占用默认端口 8554，后启动的进程会立即退出。
Start-Process -WindowStyle Minimized mediamtx.exe -ArgumentList "mediamtx_8554.yml"
Start-Process -WindowStyle Minimized mediamtx.exe -ArgumentList "mediamtx_8555.yml"
Start-Process -WindowStyle Minimized mediamtx.exe -ArgumentList "mediamtx_8560.yml"

# 步骤 2: 等 3 秒让 RTSP 服务器就绪
Start-Sleep 3

# 确认三个端口都已监听；应分别看到 8554、8555、8560
Get-NetTCPConnection -State Listen |
    Where-Object { $_.LocalPort -in 8554, 8555, 8560 } |
    Select-Object LocalPort, OwningProcess |
    Sort-Object LocalPort

# 步骤 3: 启动采集模块
.\ingest-streaming-module.exe
```

MediaMTX 不会由 `ingest-streaming-module.exe` 自动启动。它是 A 与 B/C 之间的 RTSP 中转服务：

| 实例配置 | 端口和路径 | 用途 |
|----------|------------|------|
| `mediamtx_8554.yml` | `rtsp://127.0.0.1:8554/main` | A 发布 cam_01，B/C 拉取主机位 |
| `mediamtx_8555.yml` | `rtsp://127.0.0.1:8555/aux` | A 发布 cam_02，B/C 拉取辅机位 |
| `mediamtx_8560.yml` | `rtsp://127.0.0.1:8560/program` | B 发布导播预览流 |

如果上述三个端口已经处于监听状态，说明 MediaMTX 已运行，不要重复启动。只有在端口未监听时才执行三条 `Start-Process` 命令。

**验证 A 启动成功**（另开终端）:
```powershell
# 检查 RTSP 流
& "I:\GitHub\football-auto-broadcast-\third_party\windows\ffmpeg\bin\ffprobe.exe" -rtsp_transport tcp rtsp://127.0.0.1:8554/main
& "I:\GitHub\football-auto-broadcast-\third_party\windows\ffmpeg\bin\ffprobe.exe" -rtsp_transport tcp rtsp://127.0.0.1:8555/aux
# 应显示 1920x1080 @ 25fps H.264

# 检查 A 状态
curl http://127.0.0.1:8081/api/v1/ingest/status
```

**预计行为**:
- 启动后立即在 8083 端口提供 HTTP 服务
- 收到 `/matches/start` 后启动 FFmpeg 子进程拉流（约需 2-5 秒建立连接）

## 4. 快速 E2E 验证（命令行）

全部模块启动后，在任意终端执行：

```powershell
# E2E 脚本 — 在任意 PowerShell 终端执行（需 A/B/C/D/E 全部启动）

# 1. 创建比赛（match_id 由 E 自动生成）
$r1 = Invoke-RestMethod -Uri "http://127.0.0.1:8080/api/v1/matches" -Method POST -Body '{"match_name":"E2E Test"}' -ContentType "application/json"
$mid = $r1.data.match_id
Write-Host "Match: $mid"

# 2. 开始比赛（E 自动编排 A→B→C 初始化+启动）
$r2 = Invoke-RestMethod -Uri "http://127.0.0.1:8080/api/v1/matches/$mid/start" -Method POST -ContentType "application/json"
if ($r2.code -ne 0) { Write-Host "START FAILED: $($r2.data.init_results)" -ForegroundColor Red; return }
Write-Host "Start OK: $($r2.data.status)"

# 3. 等待 30 秒（让系统充分运行，C 模块 FFmpeg 连流 + YOLO 推理）
Write-Host "Waiting 30s..."
Start-Sleep 30

# 4. 检查 C 模块状态（关键验收点）
$c = Invoke-RestMethod "http://127.0.0.1:8083/api/v1/vision/matches/$mid/status"
Write-Host "C: status=$($c.data.status) cam01=$($c.data.camera_main_status) cam02=$($c.data.camera_aux_status) events=$($c.data.events_detected)"

# 5. 检查系统全貌
$s = Invoke-RestMethod "http://127.0.0.1:8080/api/v1/system/status"
Write-Host "System: ingest=$($s.data.ingest_status) record=$($s.data.record_status) vision=$($s.data.vision_status)"

# 6. 停止比赛（E 自动编排 C→B→A 停止）
$r3 = Invoke-RestMethod -Uri "http://127.0.0.1:8080/api/v1/matches/$mid/stop" -Method POST -ContentType "application/json" -TimeoutSec 20
Write-Host "Stop: code=$($r3.code) status=$($r3.data.status)"

# 7. 验证产物
$raw = Test-Path "D:\football\data\raw\$mid"
$prog = Test-Path "D:\football\data\program\$mid"
$evt = Test-Path "D:\football\data\metadata\$mid\event_candidates.json"
$idx = Test-Path "D:\football\data\metadata\$mid\record_index.json"
Write-Host "raw=$raw | program=$prog | events=$evt | record_idx=$idx"

# 8. 验证无残留 FFmpeg 进程（C 模块 stop 后应清理）
$ff = @(Get-Process -Name "ffmpeg" -ErrorAction SilentlyContinue).Count
Write-Host "residual ffmpeg: $ff (B module's 2 recording ffmpegs are normal)"
```

## 6. 停止系统

**从后往前关闭**:

```powershell
# 1. 关闭 E 模块 — 在终端 4 按 Ctrl+C
# 2. 关闭 D 模块 — 关闭其最小化窗口（或 Get-Process highlight* | Stop-Process -Force）
# 3. 关闭 C 模块 — 在终端 3 按 Ctrl+C
# 4. 关闭 B 模块 — 在终端 2 按 Ctrl+C
# 5. 关闭 A 模块 — 在终端 1 按 Ctrl+C，然后关闭 3 个 MediaMTX 窗口
```

彻底清理：
```powershell
# 确保所有模块进程已停止（注意进程名与实际 exe 一致）
Get-Process -Name "ingest-streaming-module","record_program_service","vision_event_service","highlight_generation_service","platform_orchestration_service","mediamtx" -ErrorAction SilentlyContinue | Stop-Process -Force

# B 模块的录制 FFmpeg 进程（正常残留，stop 后会自行退出）
Get-Process -Name "ffmpeg" -ErrorAction SilentlyContinue | Stop-Process -Force
```

## 7. 常见问题

| 问题 | 原因 | 解决方法 |
|------|------|----------|
| A 模块 "Camera not found" | 相机序列号不匹配或未连接 | 检查 PoE 交换机，核对 `config.json` 中 serial |
| 双击 `mediamtx.exe` 后窗口闪退 | 默认配置尝试占用 8554；该端口已被已有实例占用，或没有按实例指定配置 | 不要直接双击；按终端 1 的三条 `Start-Process` 命令启动，并先检查 8554/8555/8560 是否已监听 |
| MediaMTX 端口被占用 | 上次未正常关闭 | `netstat -ano \| findstr ":8554"` 查看并 kill 占用进程 |
| 8080 API 正常但平台首页返回 HTTP 404 | `build/bin` 缺少 `web` 或 `configs` 运行资源 | 重新构建 E；CMake 会自动复制这两个目录到 EXE 同级。确认 `build/bin/web/index.html` 存在后重启 E |
| **B 模块 stop 超时 / E stop 返回 1007** | MediaMTX 8560 未启动，B 的 preview pipe 阻塞 | **启动第三个 MediaMTX 实例 (8560)**，见终端 1 启动命令 |
| **C 模块启动后无法加载配置/模型** | 工作目录不正确 | **必须从 `build-ninja-msvc` 目录启动**，不能从仓库根目录启动 |
| C 模块 `status=degraded` | FFmpeg 子进程还在连流（首次需 2-5 秒）或 RTSP 异常 | `t+5s` 内 degraded 正常；`t+10s` 仍 degraded 则检查 A 模块 RTSP 是否正常 |
| C 模块 `ffmpeg_path not configured` | 配置文件路径不对 | 确认 `config/default_config.yaml` 中 `ffmpeg_path` 指向有效路径 |
| Invoke-RestMethod 返回 `connection refused` | 对应模块未启动 | 检查该模块终端是否有错误日志 |
| 录像文件不存在 | B 模块未收到 start 消息或磁盘满 | 检查 B 模块日志，确认 `data_root` 磁盘空间 |
| **E stop 代码 1007，但实际模块已停止** | B 的 `program_worker.join()` 超时，E 等待超时 | 查看各模块状态确认实际已停止；重新启动 B 即可恢复 |
| 关闭后仍有 ffmpeg.exe 进程（>4个） | C 模块异常退出未清理子进程，或 MediaMTX 残留 | 正常情况 B 有 2 个录制进程；多余的用 `taskkill /f /im ffmpeg.exe` |
| **cam_01 online 但 cam_02 offline** | cam_02 的 FFmpeg 子进程重连中 | 等待 10-15 秒；如持续 offline 检查 `rtsp://127.0.0.1:8555/aux` 是否可播放 |
