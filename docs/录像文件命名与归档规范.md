# 录像文件命名与归档规范 v1

## 文档信息
- 版本：v1
- 依据：`Team Interface Contract v1.2 Frozen` §5（Windows 文件系统与目录契约）、§8.7（record_index.json 结构）、§10（日志与追踪约定）
- 适用模块：record-program-module（录像与主画面生成模块，模块 B）
- 消费模块：highlight-generation-module（模块 D，读取录像索引与文件）
- 适用阶段：第 1 周文件格式冻结 → 全周期
- 冻结要求：命名规则、目录结构、容器格式一旦冻结，未经团队评审不得修改

---

## 1. 数据根目录（合同 §5.1 冻结）

所有录像文件统一存放在数据根目录下，通过配置文件读取，禁止硬编码路径。

```
{data_root}\
```

Windows 正式部署推荐值（合同 §5.1）：`D:\football\data\`

---

## 2. 顶层目录结构（合同 §5.2 冻结）

```
{data_root}\
├── raw\              # 原始录像（必保留）
├── program\          # 裁切与主画面录像
├── output\           # 集锦输出
├── metadata\         # 元数据（索引、事件、关注区域）
├── logs\             # 运行日志
└── temp\             # 临时文件（可清空）
```

---

## 3. 原始录像 (raw)

### 目录结构

```
{data_root}\raw\{match_id}\
```

### 文件命名

| 文件 | 路径 | 说明 |
|------|------|------|
| 主机位原始录像 | `{match_id}\cam_01.mp4` | 半场边线主机位，全场原始视频 |
| 辅机位原始录像 | `{match_id}\cam_02.mp4` | 底线角区辅机位，全场原始视频 |

### 录像参数

| 参数 | 冻结值 | 说明 |
|------|--------|------|
| 容器格式 | `.mp4` | H.264 编码的 MP4 容器 |
| 视频编码 | H.264 | 由 ingest 模块输出，录制时不解码再编码 |
| 分辨率 | 2592×1944 → 1920×1080 | 输入为 5MP 原生，输出按管道缩放至 1080p |
| 帧率 | 25 fps | 固定 |
| 像素格式 | YUV 4:2:0 (I420) | H.264 编码后的标准输出 |

### match_id 命名规则

格式：`match_{YYYYMMDD}_{序号}`

示例：
```
match_20260405_001    # 2026年4月5日第1场比赛
match_20260405_002    # 2026年4月5日第2场比赛
```

- `YYYYMMDD`：比赛日期（8位数字）
- 序号：3位数字，从 001 递增，同一天内不重复

## 4. 主画面录像 (program)

### 目录结构

```
{data_root}\program\{match_id}\
```

### 文件命名

| 文件 | 路径 | 说明 |
|------|------|------|
| 主画面录像 | `{match_id}\program.mp4` | 经过裁切/导播后的输出主画面 |
| 主机位裁切 | `{match_id}\cam_01_cut.mp4` | 主机位视线裁切结果（可选保留） |
| 辅机位裁切 | `{match_id}\cam_02_cut.mp4` | 辅机位视线裁切结果（可选保留） |

### 主画面输出规格

| 参数 | 冻结值 | 说明 |
|------|--------|------|
| 容器格式 | `.mp4` | H.264 编码的 MP4 容器 |
| 分辨率 | 1920×1080 | 固定 1080p |
| 帧率 | 25 fps | 固定 |
| 宽高比 | 16:9 | 裁切策略保证输出 16:9 |
| 裁切源 | cam_01 / cam_02 | 根据视觉模块的多机位决策动态切换 |

### 保留策略

| 文件类型 | 保留策略 | 说明 |
|------|------|------|
| 原始录像 (cam_01/02) | 必保留 | 赛后生成集锦和回查的唯一来源 |
| 主画面录像 (program.mp4) | 默认保留 | 快速回看 |
| 裁切录像 (cam_*_cut.mp4) | 默认保留 | 磁盘不足时优先清理 |

## 5. 元数据文件 (metadata)

### 目录结构

```
{data_root}\metadata\{match_id}\
```

### 文件命名

| 文件 | 路径 | 说明 |
|------|------|------|
| 录像索引 | `{match_id}\record_index.json` | 录像文件路径与时间统计 |
| 候选事件 | `{match_id}\event_candidates.json` | 视觉模块输出的候选事件 |
| 关注区域 | `{match_id}\focus_regions.json` | 双机位关注区域时间序列（可选落盘） |
| 多机位决策 | `{match_id}\program_decision.json` | 机位选择决策日志（可选落盘） |

```json
{

  "record_end_timestamp_ms": 1712328600000,
  "duration_sec": 5400,
  "video_codec": "h264",
  "container": "mp4",
  "fps": 25,
  "source_resolution": "5mp_native",
  "program_resolution": "1920x1080",
  "cam_01_raw_path": "{data_root}\\raw\\match_20260405_001\\cam_01.mp4",
  "cam_02_raw_path": "{data_root}\\raw\\match_20260405_001\\cam_02.mp4",
  "cam_01_cut_path": "{data_root}\\program\\match_20260405_001\\cam_01_cut.mp4",
  "cam_02_cut_path": "{data_root}\\program\\match_20260405_001\\cam_02_cut.mp4",

}
```

| 参数 | 冻结值 | 说明 |
|------|--------|------|

## 7. 日志文件（合同 §10 冻结）

合同 §10：所有日志必须包含 `match_id`；涉及机位的日志必须包含 `camera_id`；涉及多机位决策的日志必须包含 `recommended_camera_id` 与 `reason`。

### 目录结构

```
{data_root}\logs\
```

### 文件命名（合同 §10.4 冻结）

```
{module_name}_{YYYYMMDD}.log
```

### 日志内容要求

| 字段 | 要求 | 合同依据 |
|------|------|------|
| match_id | 所有日志必须携带 | §10.1 |
| camera_id | 涉及机位的日志必须携带 | §10.2 |
| recommended_camera_id + reason | 涉及多机位决策的日志必须携带 | §10.3 |
| 时间戳 | 毫秒级系统时间戳 | §7.2 |
| 级别 | INFO / WARNING / ERROR / DEBUG |

| 目录 | 创建时机 | 创建者 |
|------|------|------|

| `metadata/` | 初始化 | 平台模块 |
| `logs/` | 初始化 | 平台模块 |

| `{match_id}/` | 创建比赛时 | 平台模块 |
| raw 子目录 | 开始比赛时 | 录像模块 |
| program 子目录 | 开始比赛时 | 录像模块 |
| output 子目录 | 集锦生成前 | 集锦模块 |

---

## 9. 强制要求（合同 §5.4、§5.5）

1. 所有路径必须从配置文件读取，禁止硬编码（合同 §5.5.1）
2. 禁止模块创建第二套根目录（合同 §5.5.3）
3. 写文件后必须先关闭文件句柄再更新索引（合同 §5.5.4）
4. 开始比赛前必须检查磁盘剩余空间（合同 §5.4），E 模块必须告警
5. 同一 match_id 下的目录结构不得在比赛中途变更
6. 删除或清理文件必须通过 E 模块统一操作
7. 存储预算：当前 MVP 可用磁盘 ~300GB（合同 §5.4）；双路原始录像必保留，主画面默认保留；低于安全阈值时 E 模块告警
