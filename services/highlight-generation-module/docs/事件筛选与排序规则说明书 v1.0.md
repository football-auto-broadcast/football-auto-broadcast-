# 事件筛选与排序规则说明书 v1.0

**文档版本**: v1.0  
**最后更新**: 2026-05-20  
**适用模块**: `highlight-generation-module` (模块 D 内部核心算法规约)  
**适用架构**: Windows 原生单主机集中式架构  
**输入源**: `vision-event-module` 输出的 `events.json`

---

## 1. 文档目的

本文档用于规范模块 D 在收到视觉分析模块（C 模块）生成的原始候选事件集（Candidate Events）后，如何进行数据清洗、置信度过滤、时域去重重叠合并，以及高光片段的优先级排序。本规范旨在确保生成的全场精彩集锦在有限时长内包含最具观赏性的画面，避免镜头冗余与时间轴冲突。

---

## 2. 核心筛选机制

模块 D 从 `configs/highlight-generation-module/events.json` 读取事件数组后，必须执行两层物理过滤。

### 2.1 置信度阈值过滤 (Confidence Thresholding)
由于视觉模块在复杂光照或激烈对抗下可能产生虚警，模块 D 必须根据事件类型实施差异化置信度清洗。低于以下阈值的候选事件将被直接丢弃：

| 事件类型 (`event_type`)   | 最低置信度阈值 (`confidence`) | 过滤策略说明                                 |
| :------------------------ | :---------------------------- | :------------------------------------------- |
| `goal_candidate`          | `0.70`                        | 进球事件容错度高，适当放低阈值防止漏检       |
| `shot_candidate`          | `0.75`                        | 射门候选较多，提升阈值保证回放质量           |
| `danger_attack_candidate` | `0.80`                        | 危险进攻判定主观性强，采取严格的高置信度准入 |
| `celebration_candidate`   | `0.80`                        | 庆祝动作识别极易与普通跑动混淆，执行强过滤   |

### 2.2 时域边界合法性校验
对于通过置信度过滤的事件，其时间参数必须通过以下硬性校验：
1. `start_sec` 必须 $\ge 0$ 且小于 B 模块索引中声明的 `total_duration_seconds`。
2. `end_sec` 必须 $> start_sec$。
3. 单个事件的持续时间（`end_sec - start_sec`）必须在 $[2.0, 60.0]$ 秒之间，超出此区间的事件判定为视觉模块时序异常，直接剔除。

---

## 3. 事件去重与时域合并算法 (Overlap Merging)

足球比赛中，多个视觉事件往往在时间轴上高度重叠（例如：射门后立刻发生进球，紧接着触发庆祝）。模块 D 必须实现以下时域合并状态机，防止同一段视频被重复剪辑。

### 3.1 同类型事件交叉重叠合并
若同类型的两个事件 $E_1$ 和 $E_2$ 满足以下条件：
$$\max(E_1.start\_sec, E_2.start\_sec) \le \min(E_1.end\_sec, E_2.end\_sec)$$
即两段剪辑窗口存在交集，则必须将其合并为一个全新的事件：
- `start_sec` = $\min(E_1.start\_sec, E_2.start\_sec)$
- `end_sec` = $\max(E_1.end\_sec, E_2.end\_sec)$
- `confidence` = $\max(E_1.confidence, E_2.confidence)$

### 3.2 异类关联事件时序融合（智能拼接规则）
针对特定的高光链条，模块 D 执行以下跨事件类型融合逻辑：
1. **进球与庆祝融合**：若 `celebration_candidate` 的 `start_sec` 落在 `goal_candidate` 的 `end_sec` 后 30 秒以内，则自动将两事件视作“连续高光”。更新进球事件的 `end_sec` 为该庆祝事件的 `end_sec`，随后将该庆祝事件从队列中移除。
2. **射门与进球吞噬**：若 `shot_candidate` 之后 10 秒内引发了 `goal_candidate`，由于进球本身已包含完整的射门前奏，直接删除该 `shot_candidate`，保留 `goal_candidate`。

---

## 4. 排序与权重机制 (Ranking & Weighting)

为应对长时长比赛中高光事件过多的情况，系统支持集锦总容量控制（默认上限为 180 秒）。当过滤合并后的总时长超出阈值时，必须按照以下权重公式对事件队列进行降序排列，取前 $N$ 个事件至填满容量：

### 4.1 权重计算公式
每个事件的最终排位分数 $S$ 计算如下：
$$S = \text{Base\_Weight}(\text{event\_type}) \times \text{confidence}$$

### 4.2 类型基础权重分配表
| 优先级排名   | 事件类型字符串表示 (`event_type`) | 基础权重 (`Base_Weight`) |
| :----------- | :-------------------------------- | :----------------------- |
| **1 (最高)** | `goal_candidate`                  | `100.0`                  |
| **2**        | `shot_candidate`                  | `70.0`                   |
| **3**        | `danger_attack_candidate`         | `50.0`                   |
| **4 (最低)** | `celebration_candidate`           | `30.0`                   |

## 5. 校验规则与输出示例

### 5.1 内部清洗后输出格式示例
清洗和排序完成后，模块 D 内部内存或临时缓存中生成的 `filtered_events.json` 必须符合以下规范：

```json
{
  "match_id": "bupt_match_20260519",
  "total_filtered_count": 2,
  "final_highlight_duration_sec": 32.5,
  "events": [
    {
      "rank_position": 1,
      "event_id": "evt_0001",
      "event_type": "goal_candidate",
      "start_sec": 918.0,
      "end_sec": 938.5,
      "calculated_score": 95.0
    },
    {
      "rank_position": 2,
      "event_id": "evt_0003",
      "event_type": "shot_candidate",
      "start_sec": 1420.0,
      "end_sec": 1432.0,
      "calculated_score": 61.6
    }
  ]
}
```

### 5.2 边界校验约束

- **空事件处理**：若过滤后有效事件数等于 0，主程序必须立刻中断后续剪辑流程，终止代码，向系统抛出状态码 `1005`。
